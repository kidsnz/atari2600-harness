# Litmus Test 結果 — 横位置（欠落 B の実測検証）

ハーネス（`cmd/harness`）経由で player0 を位置決めし、`read_tia` の `ResetPixel` /
`HmovedPixel` を**数値で**読んだ実測値。過去 Pong の失敗 #1「魔法定数の総当たり」を
二度とやらないための、権威ある測定。**全て推測でなく実機（Gopher2600 nightly）の出力。**

ROM: `roms/litmus_pos.asm`（粗調整ループ 1 反復 = `SBC#1`+`BCS` = 5 CPU サイクル = 15 カラークロック）。
手順: `$80`(DELAY) を poke → `step_frame` → `read_tia`。HMOVE は HMCLR で 0。

## 確定事実

1. **粗調整は完全線形:** DELAY 3〜11 で `ResetPixel = 15·DELAY − 18`。
   1 ループ反復 = 5 CPU サイクル = 15 カラークロック = **15px**。**3px / CPU サイクル**を実機確認。
2. **可視幅 160 で折返し**（mod 160）。DELAY=12 が 2 に巻き戻る。
3. **最左クランプ = X=3**（プレイヤー）。CLAUDE.md の定数と一致。
4. **`HmovedPixel == ResetPixel`**（HMCLR で動きレジスタ 0・HMOVE 未発火のため）。
   → HMOVE を打つと差が出るはず（次段で検証）。
5. **座標系:** `ResetPixel`/`HmovedPixel`/ビーム可視 `Clock` は同一の可視ピクセル座標 0–159。

## CLAUDE.md 公式との関係

CLAUDE.md: プレイヤー `X = 3N − 54`（N = 同期点〜RESPx ストローブの CPU サイクル数）。
本測定のオフセット −18 は本 ROM 固有のプロローグ（HMCLR/LDA/SEC/末尾 SBC・BCS・STA）の
サイクル数を含むため定数が違うが、**傾き 3px/サイクルは一致**。重要なのは
「走らせれば数値で確定できる」＝欠落 B が埋まったこと。N の絶対値は kernel 依存で**実測する**もの。

### 自動再現（B-4, v0.20.0）

上記の手作業スイープは `cmd/calibrate` で**自動再現**できるようになった（`internal/calibrate`）。
`go run ./cmd/calibrate` が litmus_pos の DELAY を掃引し、折返し・左端飽和を頑健に除外して直線回帰
→ **slope 3.0000 px/CPU-cycle・offset −18・R²=1.0** を出力（本表と一致）。kernel パターンを変えたら
協調 ROM を差し替えて再較正すれば、その kernel のオフセット定数を毎回数値で得られる。

### 確定事実（HMOVE）

6. **HMOVE ニブル表は CLAUDE.md と完全一致**（全 16 値）。上位ニブルのみ・2 の補数・
   **正=左（X 減）/ 負=右（X 増）**・範囲 **+7（左7, $70）〜 −8（右8, $80）**。
7. **1px 粒度**で動く（微調整成立）。`ResetPixel` は不動（72）＝ RESP0 と HMOVE は独立。
8. 粗調整（15px）＋ HMOVE（1px）で**任意の X を完全に指定可能**。鉄則#4 合格。

## litmus test 結論

**ハーネスは本物。** 横位置を「粗 15px ＋ 微 1px」で数値的に予測・配置・検証できる。
過去 Pong の失敗 #1（魔法定数の総当たり）と #3（位置決めの破綻）は、本ハーネス上では
`read_tia` の数値で常に裏が取れる。欠落 B は埋まった。
