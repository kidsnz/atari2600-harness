# research-w9-real-kernel-patterns — 実ゲームのヘテロゾーンカーネル：M3 codegen への設計示唆

目的: M3「ゾーン合成型カーネル生成」の codegen 設計を実ゲームの逆アセンブルで裏打ちする。
手元の素材 = `reference/disassemblies/bjars_site_archive/public_html/source/`（RiverRaid・Pitfall・Defender・Adventure・Carnival）。
クリーンルーム作法厳守：バイト列・コードの転載なし。技法を一般化した散文で記し、ゲーム名＋ラベル名を breadcrumb として添える。
調査日 2026-06-13。**[確認]** = ソース実読 / **[推定]** = 未確認の一般化。

## 1. ゾーン分割カタログ（実ゲーム別）

### 1-A. 垂直2分割型 — プレイエリア上 ＋ ステータス帯下

**代表例**: RiverRaid (Activision 1982, T.Jentzsch 解析) [確認: `RiverRaid.asm` ヘッダコメント + `DisplayState SUBROUTINE` (行 968)]

RiverRaid の可視フレームは上 160 ライン（プレイエリア）＋下「StateDisplay」（スコア8行 × DIGIT_H + 燃料帯 + 残機・コピーライト帯）の 2 ゾーンに分割されている。

- **ゾーン1（メインカーネル, 上 160 ライン）**：2ライン毎に P0（ジェット）＋ P1（敵オブジェクト、VDELバッファ利用）＋ PF（川の形状）を描く。`lineNum` カウンタが 160 → 0 方向で減算され、0 になった瞬間に `JMP DisplayState` でゾーン2へ遷移 [確認: 行 794]。
- **ゾーン2（StateDisplay, 下帯）**：直前の最終 WSYNC + HMOVE で境界ラインを確定し、そのままスコア表示専用の6ポインタ引き回しループへ移行。色レジスタ (COLUBK, COLUPF, NUSIZ0/1) を一気に書き換えてスコア帯の視覚を確定 [確認: 行 968-999]。

**切替の手順**（一般化）：
1. プレイエリア最終ラインで WSYNC → HMOVE → GRP0/GRP1 を 0 クリア。
2. PF0/1/2 を 0 クリア・REFP0/1 リセット。
3. 新ゾーンの NUSIZ・色・VDEL を一括設定（次の WSYNC 前に完了が理想）。
4. スコアループへ JMP（WSYNC は各スコアライン先頭で再取得）。

---

### 1-B. 縦多ゾーン型（9ゾーン構成）— ジャングル横断ゲーム

**代表例**: Pitfall (Activision 1982, T.Jentzsch 解析) [確認: `Pitfall.asm` ヘッダコメント + Kernel 1–9 ラベル群]

Pitfall は 1 フレームの可視領域を **9 本の専用ループ**（コメントで "nine specialized kernel loops" と明記）＋スコア/タイマ表示部に分割している。各ループの高さ（ライン数）と担当要素が異なる：

| ゾーン | 高さ | 主な要素（一般化） |
|--------|------|--------------------|
| Score/Timer 帯（上） | DIGIT_H × 2 帯 | 6桁スコア＋タイマ（P0/P1 3コピー NUSIZ 利用、VDELP0/1 でバッファ）|
| Kernel 1 | 31 ライン | ツリー上部・リアナ（Ball）・葉 PF |
| Kernel 2 | 4 ライン | リアナ＋主人公(P0)・地上物(P1) の粗位置決め |
| Kernel 3 | 21 ライン | リアナ(BL)+主人公(P0) 走査線毎色 |
| Kernel 4 | 24 ライン | リアナ終端＋主人公(P0) |
| Kernel 5 | 7 ライン | 主人公＋穴/ピット(PF2)＋地上物(P1) |
| Kernel 6 | 8 ライン | 地上物下部＋穴上端 |
| Kernel 7 | 12 ライン | ラダー上部＋主人公(P0) |
| Kernel 8 | 15 ライン | 主人公＋ラダー(BL)＋壁(P1) 走査線毎色 |
| Kernel 9 | 16 ライン | 主人公＋サソリまたは壁下部(P1)＋ラダー(BL) |
| Copyright 帯（下） | 16 ライン | テキスト（P0/P1） |

[確認: 行 529/591/663/695/751/774/1084/1168/1204 の各「Kernel N」コメント]

**切替のメカニズム**：各ゾーンの末尾は WSYNC → HMOVE の後、**次ゾーンの初期設定コードをインラインで実行**してから次ループ先頭に fall-through する（`JMP ContKernel` 1箇所を除き、ほぼ直続き）。NUSIZ・CTRLPF・REFP の切替はその初期化ブロックで行う [確認: 行 562-590, 826-826]。

### 1-C. 3分割型 — レーダー帯上 ＋ プレイエリア ＋ スコア/HUD

**代表例**: Defender (Atari 1982, Bensema+Polik 解析) [確認: `Defender.asm` 行 1050-1309]

Defender は上から：
1. **レーダー帯**（約 25 ライン：1ライン枠 + 11 × 2ライン内部 + 1ライン下枠）：PF でレーダーフレームを描き、Missile0/1 で 2 つのレーダー輝点を Y 位置比較で出す [確認: 行 1072-1095]。
2. **プレイエリア**（残り大半）：P0/P1 と PF（都市景観）を走査線毎に更新する深いループ [確認: 行 1124-1260]。
3. **スコア/HUD 帯**（下部）：RESP0/1 再配置 + NUSIZ で3コピー → 残機アイコン等を描く [確認: 行 1262-1302]。

**切替の特徴**：レーダー → プレイエリアの境界では WSYNC 後にすぐ PF0/1/2 を 0 クリアし、CTRLPF を変更（CTRLPF $F0 = PF優先・反射）、その後 RESP0/1 + HMOVE でプレイヤーを再配置してから本体ループへ入る [確認: 行 1115-1224]。レーダー帯と本体ループで **P0/P1 を全く別用途**に使い回している点が重要（レーダー内 = M0/M1 でドット、本体 = P0/P1 で複数スプライト）。

---

### 1-D. 均質ループ型 — 1カーネルで全走査線を均一に回す

**代表例**: Adventure (Atari 1979) [確認: `Adventure.asm` `PrintDisplay` SUBROUTINE 行 53-194]

Adventure はゾーン分割を持たない。`PrintDisplay` → `PrintPlayer01` → `PrintPlayer00` という 3 関数が互いに BPL/BEQ で飛び合うラウンドロビン構造で全可視ライン（`$8E` カウンタ = $68 = 104 dec → 8 まで）を回す。16 ライン毎に PF0/1/2 をテーブルから引き直す（部屋定義テーブルが3バイト×行数）。

これは「全ゾーンで同一要素構成」の均質カーネルで、**M3 の目標（ヘテロ合成）とは対極**。ただし「部屋 → 別 PF テーブルポインタ」という状態別カーネルの萌芽があり、design-principles §「状態=1個の GameState + 状態別カーネル」と合致する。

---

### 1-E. 得点帯専用サブルーチン型

**代表例**: Carnival (Stern 1982) [確認: `carnival.asm` 行 341-364]

Carnival のスコアは独立したループとして組まれており、走査線の最初に WSYNC + HMOVE、その後すぐ 6桁ポインタ引き回し（P0/P1 で各2桁ずつ VDEL バッファ利用）を行う。スコア帯を抜けた後、HMCLR → VDELP0/1 → GRP0/GRP1 クリアを走らせてから次の描画ゾーン（パイプ・旗）の初期化に移行する [確認: 行 364-403]。

Pitfall の `ShowDigits SUBROUTINE`（行 828）も同型：WSYNC → HMOVE → NUSIZ/RESP/VDEL セットアップ → 8 ライン分の 6桁ポインタループ → RTS。どちらも「スコア帯はサブルーチンに切り出し、メインカーネルから JSR する」方式。

## 2. ゾーン境界の遷移技（一般化）

### 2-A. WSYNC 終端 + 即 HMOVE パターン（全例に共通）

実ゲームのほぼすべての境界で：

```
sta WSYNC    ; ← 現ゾーン最終ライン終端を確定
sta HMOVE    ; ← 水平移動を最初のサイクルで適用
```

このペアがゾーン境界の基本単位。HMOVE は WSYNC 直後の最初の命令として必ず発行する（CLAUDE.md 「HMOVE は WSYNC 直後」）。その後の 24cy 以内に HMxx を書かない制約を守るため、境界の初期化コードは HMxx を後回しにして NUSIZ / COLUBK / COLUPF / CTRLPF / GRP クリアを先に処理する。

[確認: RiverRaid 行 971/1010/1036/1088、Pitfall 行 454/485/504/519/548 など]

### 2-B. RESP 再配置とラインカウントによる精密配置

プレイエリアからスコア帯への切替時、スコア数字を等間隔に並べるために P0/P1 を **NUSIZ THREE_COPIES（$03 または $10|THREE_COPIES）+ RESP0/1 の即配置** で水平解決する。これはプレイエリアの RESP 設定を捨てて上書きするため、「プレイエリアの最終ライン終了を確認してから RESP を書く」順序が必須。

- Pitfall `ShowDigits`: WSYNC → HMOVE → NUSIZ0/1 を THREE_COPIES に、RESP0 を基準位置に発行し、HMP1 で P1 をオフセット [確認: 行 838-851]。
- RiverRaid `DisplayState`: RESP0/RESP1 を同じサイクルで書き（CTRLPF も同時）、HMCLR → NUSIZ0/1 を THREE_COPIES に [確認: 行 985-1003]。

### 2-C. VDEL フリップ（バッファ切替）

P0/P1 を VDEL 二重バッファで使うゾーンからそうでないゾーンへ切替えるとき：

- **有効化**（VDEL オン）: `STY VDELP0` (Yが $07 など非ゼロ) で次ラインで GRP が反映。スコア表示ゾーンに入るときに使う（Pitfall 行 850-851、RiverRaid 行 394/1001）。
- **解除**（VDEL オフ）: `STX VDELP0` (Xが 0) で即時反映に戻す。プレイエリアに戻るとき。

解除する際は「GRP0 を先に書いて GRP1 も書く → VDEL を 0 に」という順序で古いバッファ内容を確実に flush する（空きラインが必要な理由）。

### 2-D. NUSIZ / CTRLPF / REFP の一括切替

ゾーン境界の WSYNC 後の「初期化ブロック」の典型的な順序（HMOVE 直後から 24cy 制約が終わる間）：

1. GRP0/GRP1 を 0 クリア（表示内容を即消去）
2. PF0/1/2 を 0 クリア（前ゾーンの PF を消す）
3. COLUBK / COLUPF を新ゾーンの色に（視覚的な帯の区切り感を演出）
4. NUSIZ0/1 を新サイズに
5. CTRLPF を新反射/優先度に
6. REFP0/1 を新方向に
7. （24cy経過後）HMxx を書いて次の WSYNC 前に HMOVE の移動量を仕込む

[確認: RiverRaid 行 976-1003（項目3〜6の典型）、Pitfall 行 825-848（项目1〜4）、Defender 行 1115-1224（全手順が明示）]

### 2-E. 76cy と HMOVE 後 24cy 制約の実際の守り方

観測されたパターン（一般化）：
- スコア帯ループは「WSYNC → HMOVE → [ポインタ引き + GRP0/1 書込みで 76cy ちょうど]」で 1 ライン完結させ、HMOVE 後の HMxx 書込みを一切行わない（HMCLR も使わない）。
- HMOVE が必要なゾーン（Ball を動かすリアナ描画など）は HMOVE 直後の数命令を「HMxx 禁止ゾーン」として nop/bit などのスプレッダで埋め、24cy 経過後に HMBL 等を書く [確認: Pitfall 行 533-539 の HMBL 書込みシーケンス]。
- 境界の RESP 再配置は「次の WSYNC 後 → HMOVE → NOP列 → RESP → さらに WSYNC → HMOVE」という 2ライン消費パターンが多用される [確認: Pitfall 行 502-519、Defender 行 1212-1221]。

## 3. 要素の使い回し（P0/P1 の複数ゾーン別用途）

### 3-A. P0 = スコア桁 / プレイ時 = キャラクタ

最も頻出のパターン。スコア帯では P0 と P1 を NUSIZ THREE_COPIES で 6 桁の数字スプライトに充て、プレイエリアではそれを解放して主人公＋敵に割り当てる。

- Pitfall: スコア帯（JSR ShowDigits 内）で `NUSIZ0 = $10|THREE_COPIES`、プレイカーネルで `NUSIZ0 = ONE_COPY`、kernel 8/9 では `NUSIZ1 = ONE_COPY` [確認: 行 838/586]。
- RiverRaid: DisplayState で `NUSIZ0 = THREE_COPIES`、プレイカーネルでは NUSIZ1 を各ブロックの State1Lst から取得（敵の大きさに応じて可変）[確認: 行 349/995]。
- 共通して: RESP0/RESP1 はゾーン初期化ブロックで毎回上書きされる。スコア帯の配置とプレイ帯の配置は完全に別物として管理。

### 3-B. P1 = レーダードット / プレイ時 = 敵スプライト

Defender では P1 のカラーレジスタをレーダー帯で `radarColor`（レーダー輝点色）に設定し、プレイエリアに入った時点で `gameColors+N` に書き換える。TIA の COLUP1 を毎ゾーン先頭に 1 命令で上書きするだけで用途が切替わる [確認: 行 1055-1056/1113-1114]。

### 3-C. Missile → Ball の帯別切替

Pitfall のリアナは Ball（ENABL）で描かれる（BL=縦細い線が liana に最適）。地下帯（Kernel 7-9）ではラダーに ENABL を使い続け、最終出口で ENABL = 0 に解除してから次フレームへ進む [確認: 行 1107/1201/1235-1238]。Ball をプレイエリアの装飾（リアナ）と HUD 要素（ラダー形状の一部）の両方に使い回す例。

## 4. M3 codegen への設計示唆（w5 を実例で補強・修正）

### 4-A. w5 §4-B「ゾーン合成型」の実例による確認

**w5 の推奨アーキ**（ゾーン列 = 断片連結 + 境界は生成器が縫う）は、実ゲームで正確に採用されている：

- Pitfall: 9 つの "specialized kernel loops" は「各ゾーン = 固定ループ断片」そのもの。境界はインライン初期化ブロック（固定命令）で縫われており、w5 §4-B の「ゾーン間縫合コードは生成器が吐く」と正確に対応する。
- RiverRaid: `DisplayState SUBROUTINE` 全体が「プレイエリア → スコア帯」の境界縫合ルーティン。この縫合を関数に切り出している点が注目で、**M3 の境界縫合テンプレも同様に「前ゾーン要素 → WSYNC/HMOVE → 後ゾーン初期化 → 後ゾーン先頭」という定型シーケンスとして生成**できる。

**w5 への補強 [推定]**：
- ゾーン境界縫合の「型」は **2 ライン消費が頻出**（1ライン目 = WSYNC+HMOVE+GRP/PF クリア、2ライン目 = WSYNC+HMOVE+新設定）。1ライン以内に全切替を収める場合は GRP クリアと色設定だけに絞り NUSIZ は次ラインに回す。codegen は「縫合ライン数の予算 = 1 or 2」を境界ノードのパラメータとして持つべき。

### 4-B. 境界縫合の自動生成テンプレ（一般化）

実ゲームの観察から抽出した境界縫合の**最小テンプレセット**：

**型 I — クリアのみ（0追加ライン・前ゾーン最終行内処理）**
前ゾーンの最終 WSYNC ループ内で GRP0/GRP1 = 0 を挿入し、新ゾーンのカラーを先読みして STA COLUBK / COLUPF。PF と NUSIZ は変えない（同一構成が続くゾーン間）。

**型 II — 色・NUSIZ 切替（1追加ライン）**
WSYNC → HMOVE → GRP/PF クリア → NUSIZ/CTRLPF/COLUBK/COLUPF 設定 → VDEL 設定。次ゾーンの先頭ループに直続き。RESP は変えない。

**型 III — RESP 再配置付き（2追加ライン）**
1ライン目: WSYNC → HMOVE → GRP/PF クリア → HMxx 設定（24cy後）。
2ライン目: WSYNC → HMOVE → RESP0/RESP1 発行（所定サイクルでストローブ）→ NUSIZ/色 設定。
[例: スコア帯への移行 = Pitfall ShowDigits 行 843-851、RiverRaid DisplayState 行 984-999]

**型 IV — HMOVE+RESP 同時（2ライン、リポジションとHMOVE微調整あり）**
HMOVE ラインで粗位置を確定し、次ラインで HMP0/1 を設定して2回目の WSYNC → HMOVE で微調整。[例: Pitfall Kernel 1 入口 行 502-520]

codegen は前後のゾーン要素集合（変化する TIA オブジェクト集合）を diff して型 I/II/III/IV を自動選択する。

### 4-C. ゾーン分割粒度の実例による指針 [確認]

**最小ゾーン高さ**: Pitfall Kernel 2 = 4 ライン。これ以下は RESP 再配置の 2 ライン消費で潰れる（[推定]: RESP 再配置を含む境界は 2 ライン以上の高さを要する）。

**最大ゾーン高さ**: RiverRaid プレイエリア = 160 ライン（2 ライン反復 × 80 + 8 ライン特殊）。1ループで全高さを吸収し内部で `lineNum` カウンタで終端判定。

**codegen の分割規則への影響**：w5 §5-b ①「ゾーン分割器: 要素構成が一定の帯に切る」の実装は、**4ライン未満の帯は隣接ゾーンに吸収する**という安全ルールを持つべき [推定]。

### 4-D. 状態別カーネル（GameState × ゾーン構成）

Pitfall はシーン型（`sceneType` 変数）によって Kernel 7-9 の内部データポインタが変わる（`ladderFlag` / `wallPatPtr` / `undrPatPtr` で切替）。同じゾーン 7-9 の「断片テンプレ」を使い回しつつ、**ポインタ（データ表）だけをシーン毎に差し替える**。

→ M3 codegen への直接示唆: ゾーン断片テンプレは「命令列テンプレ＋データ表ポインタ」の対で管理し、同一テンプレを複数シーンで再利用。w5 §5-a 「要素グラフィック層（データ）+ シーン配置層（レイアウト）」の 2 層中間表現が実ゲームで機能していることが確認された [確認 + 推定]。

### 4-E. 得点/HUD 専用サブルーチンの汎用性

Pitfall `ShowDigits`、Carnival `LF546` スコアループ、RiverRaid `.loopScore` はいずれも：
- **NUSIZ THREE_COPIES** で P0/P1 を 6 桁スコア表示に特化
- **VDELP0/1 有効化** で 1 ライン分のバッファを確保
- **走査線毎 GRP 引き回し** で桁グラフィックを表示
- **末尾に GRP/VDEL クリア**（次ゾーンに汚染しない）

という同一の型。この「スコア6桁ゾーン断片」は M4 テンプレの `score6` に対応し、最も再利用頻度が高い境界縫合パターン。w5 §6 「要確認: テンプレ断片カタログの初期セット」への回答として、**`score6` テンプレを最優先で用意**するべき [確認]。

## 5. 未解決・要確認事項

### 5-A. 境界縫合の 76cy 検証（最大の不確実性）

w5 §6 の最大未解決課題は本調査でも解消されていない。「型 III（2ライン RESP 再配置）」の命令列が実際に 76cy に収まっているかは harness の `assert_line_budget` で実走確認が必要。Pitfall の ShowDigits 入口ブロック（行 843-851）は目算でおおよそ 74cy 程度だが、**プロローグのサイクル数がカーネル入口条件に依存**するため、単独では計算できない [推定]。

### 5-B. 2ライン解像度カーネル内でのゾーン切替

RiverRaid は 2 ライン反復カーネル（奇数/偶数ラインで役割が異なる）の中でブロック境界（8 特殊ライン）を処理する。2 ライン反復ループがどの位相（奇数/偶数）でゾーン切替を行うべきかは実例から読み取ったが、**2 ライン単位の位相ロック**（境界は必ず偶数ラインから始める）という制約が正しいかどうか harness での実証が必要 [推定]。

### 5-C. VDEL バッファ flush の空きライン

VDEL を解除する際に「GRP0/1 を両方書いてから VDEL = 0」という1ラインが余分に消費される。これが「境界の型 III/IV で 2 ライン必要な理由」と関連するが、VDEL の flush コスト（何サイクル・何ライン）を codegen がどう勘定するかは設計未確定 [推定]。

### 5-D. スコア帯の高さの実測

Pitfall の DIGIT_H = 8 ライン × 2 帯（スコア＋タイマ）= 16 ライン + セパレータ = 約 18 ライン。RiverRaid は DIGIT_H = 8 ライン × 2 段（得点2プレーヤ分）+ 燃料帯 15 ライン + Copyright 帯 = 合計 40〜50 ライン。**スコア帯の正確な高さ範囲**（最小/標準/最大）を harness で実測し、M3 の「スコアゾーン高さパラメータ」の既定値を定める必要がある [要実測]。

---

## 6. 結論まとめ（M3 codegen への即用）

1. **w5 §4-B ゾーン合成型の採択は正しい**：実ゲームの全例が「固定ループ断片 + 境界初期化ブロック」構造を採用しており、テンプレ穴埋め型（§4-A）と完全生成型（§4-C）の中間の「断片連結+縫合生成」が正解だと確認された。

2. **境界縫合の型は 4 種**（I〜IV）に分類でき、codegen は前後ゾーンの TIA 要素差分から自動選択する。RESP 再配置の有無が最大のコスト差（+1ライン）。

3. **`score6` 断片テンプレが最優先**：全ゲームに共通して存在し、NUSIZ THREE_COPIES + VDEL + 走査線毎 GRP の組合せで構成。型 III 境界縫合のリファレンス実装にもなる。

4. **P0/P1 の帯別用途切替は毎ゾーン先頭での NUSIZ/COLUP/RESP 上書き**で行う。codegen はゾーン記述に「このゾーンで P0/P1 が担う用途」を持ち、前ゾーンとの diff から縫合型を決定する。

5. **最小ゾーン高さ = 4 ライン**（Pitfall 実測）。2 ライン以下はゾーンとして独立させず隣接ゾーンに吸収するルールを codegen に組む。

### 報告用1行結論

「実ゲームのヘテロゾーンカーネルは全例が "固定ループ断片 + 型 I〜IV 境界縫合" 構造を採用しており、w5 §4-B を裏打ちすると同時に、境界縫合の型分類（RESP 再配置有無 = ±1ライン）と score6 断片テンプレの最優先化という 2 点を codegen 設計に追加する。」

**最大の不確実性**: 境界縫合コード（特に型 III/IV の RESP 再配置付き 2ライン）が 76cy 制約を実際に守れるかは未実走。これは w5 §7 の「2ゾーン最小プロト」で `assert_line_budget` を使って harness で最初に潰すべき de-risk 点（変わらず最高優先）。
